## 分题型结果

| Cutoff | multi-hop | open-domain | single-hop | temporal |
|---|---|---|---|---|
| top_10 | 161 / 282 (**57.1 %**) | 47 / 96 (**49.0 %**) | 448 / 841 (**53.3 %**) | 23 / 321 (**7.2 %**) |
| top_20 | 187 / 282 (**66.3 %**) | 50 / 96 (**52.1 %**) | 499 / 841 (**59.3 %**) | 25 / 321 (**7.8 %**) |
| top_50 | 209 / 282 (**74.1 %**) | 59 / 96 (**61.5 %**) | 560 / 841 (**66.6 %**) | 44 / 321 (**13.7 %**) |
| top_200 | 237 / 282 (**84.0 %**) | 58 / 96 (**60.4 %**) | 642 / 841 (**76.3 %**) | 62 / 321 (**19.3 %**) |

**观察**：

- **multi-hop** 表现最好，top_200 达到 84.0 % — 多条事实联合推理能靠多检索补偿。
- **temporal** 是明显短板，top_200 仍只有 19.3 % — 时间敏感问题对记忆排序和时序理解要求高，单纯扩大窗口收益有限。
- **open-domain** 表现中等，扩大窗口甚至略降（61.5 % → 60.4 %），说明这部分问题靠的是"能否召回"，不是"召回多少"。

## 改进方向建议

1. **temporal 题型是主要瓶颈**（19.3 % @ top_200）
   - 强化时间感知：让 mem0 提取 facts 时附带时间戳
   - retrieval 阶段加时间近因性加权（recency boost）
   - answerer prompt 加入"对话发生时间"作为参考

2. **open-domain 在 top_50 → top_200 反而下降**
   - 说明这部分问题需要更精准的 top_k，而不是堆数量
   - 考虑 reranker（cross-encoder）精排 top_50

3. **速度优化**（如需）
   - 后期 32 s/it 主要是 DeepSeek 限流——可考虑并发降到 5 以下，或切到付费档
   - LLM client 已支持 `OPENAI_BASE_URL`，可平切到其他更快的 OpenAI 兼容服务

4. **后续可对比**
   - 用同一个数据集 + 不同 embedder（如 `BAAI/bge-m3` 1024 维）跑一遍，看 retrieval 质量
   - 用 `--predict-only` 跳 judge 阶段，单独验证 answerer 能力
